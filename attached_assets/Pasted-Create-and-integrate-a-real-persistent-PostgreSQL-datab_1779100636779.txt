Create and integrate a real persistent PostgreSQL database for the EPS (Exam Preparation System) project.

Context:
The system already has a working frontend and backend for Sprint 2, but currently some data is temporary, static, or not fully persisted.

Goal:
Implement a complete persistent database layer so all important system data is saved permanently and survives refreshes, server restarts, and new sessions.

The system must support:
- persistent users
- persistent authentication data
- persistent courses/topics/subtopics
- persistent enrollments
- persistent questions and answers
- persistent exams and exam results
- persistent notifications/messages
- persistent profile images and account data

Use PostgreSQL and connect it properly to the existing backend.

====================================
DATABASE REQUIREMENTS
====================================

Use PostgreSQL.

Use environment variable:
DATABASE_URL

Example:
DATABASE_URL=postgresql://USER:PASSWORD@HOST:PORT/DATABASE

Do not use local in-memory arrays or fake JSON storage anymore.

====================================
PERSISTENT FUNCTIONALITY
====================================

1. User Registration Persistence
When creating a new user:
- Save user permanently in database.
- Email must be unique.
- Password must be hashed using bcrypt.
- User must remain after server restart.

2. Authentication Persistence
- Login must validate against database users.
- JWT authentication remains working.
- Auth middleware must fetch real users from DB.

3. Profile Persistence
- Profile image updates must persist.
- Account updates must persist.
- Sidebar avatar must remain after refresh.

4. Course Persistence
When admin creates a course:
- Save permanently in DB.
- Course remains after restart.
- Course appears for enrolled users.

5. Enrollment Persistence
When assigning students or lecturers to courses:
- Save in enrollments table.
- Relationship must persist.
- Students must see only enrolled courses.

6. Question Bank Persistence
When lecturer:
- creates question
- edits question
- archives question
- adds answers
- changes difficulty

Everything must persist in DB.

7. Exam Persistence
When student:
- generates exam
- starts exam
- answers questions
- submits exam

Save:
- exam session
- selected answers
- correctness
- final score
- timestamps

Exam history must remain permanently.

8. Exam Review Persistence
Students must be able to:
- refresh page
- log out
- return later
- still see previous exams and results

9. Notification Persistence
Notifications must:
- belong to specific users
- persist read/unread status
- remain after refresh/relogin

10. Message Persistence
Messages must:
- persist permanently
- open correctly
- preserve read state
